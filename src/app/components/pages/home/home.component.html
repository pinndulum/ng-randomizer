Mock Randomizer Demo

Documenting @joxnathan/mock-randomizer

Synthetic data examples for UI demos, tests, and repeatable fixtures

This Randomizer UI demonstrates the @joxnathan/mock-randomizer package. Loaded as a dev dependency, mock-randomizer can replace service calls, model providers, and static fixtures in Node project tests. The examples here are meant to be references for the kinds of synthetic data that can be generated and how that data can be shaped for real use.

Random data demos

  • Buzzwords shows deterministic query params and custom word banks.
  • Shakespeare Insults combines reusable word lists with browser speech synthesis.
  • Flip A Coin visualizes random choices and aggregate outcomes.
  • Pseudo Identity assembles realistic names, addresses, phones, emails, and credit-card-shaped values.
  • Change Log History generates nested arrays and replayable seeded data.
  • Mock DSL Fill lets you experiment with the package JSON DSL.

Not so random

  • Pi Digits is a deterministic reference page for a familiar non-random sequence.
  • World Clock displays time-zone behavior without pretending the clock itself is random.

What to notice

Seeded generation matters when a failing test or demo needs to be replayed exactly. The Change Log History page is the clearest example: the same seed, template, and options regenerate the same dataset instead of merely producing similar-looking data.

What mock-randomizer generates

Random numbers, dates, booleans, strings, and other primitive values can be returned directly. Those values can then be placed inside objects and arrays. Arrays can be instantiated with a random number of elements between lower and upper limits, and each item in that array can be generated from the same template.

The package also provides realistic data rather than simply random characters. A name does not have to look like "iEwEVhrKyBYc"; it can look like "Rachel" or "Joseph". Embedded word banks provide names for people, places, things, titles, companies, and type-set copy. You can also provide your own word bank, enum, or list of choices.

How to explore it

  • Use the refresh buttons to compare fresh random output against replayed seeded output.
  • Copy generated JSON from Change Log History and inspect the nested object shape.
  • Change the seed value, refresh once, then refresh again with the same seed to see replay behavior.
  • Use the Buzzwords and Shakespeare pages to see how URL query params can preserve a generated choice.
  • Open Mock DSL Fill when you want to experiment with JSON DSL templates directly.

Realistic and valid-looking data

The Pseudo Identity page shows a single generated object with properties such as name, address, phone number, credit-card-shaped values, email, and short biography copy. Realistic data should be understandable to a human reader, but it should also be able to pass common validation checks. U.S. phone numbers, zip codes, and time zones can be influenced by state, and generated card numbers can pass Luhn checksum validation.

Generated collections and replay

The Change Log History page demonstrates filling a collection of objects. Each record contains its own generated array, and each nested change can receive previous and current value objects based on the generated change type. Seed replay makes this useful for tests: the same seed, template, and options regenerate the same dataset instead of producing a merely similar dataset.

The result of flipping a coin is deterministic if you know the side the coin started on and how many times it spun before coming to rest. In the Flip A Coin page, those considerations are randomized with a brief pause for anticipation. Results are added to a table and should trend toward a nearly 50/50 outcome after enough flips.
